# AGENTS.md

## Назначение документа

Этот файл фиксирует инфраструктурную и архитектурную "карту проекта" для работы AI-агентов и как база знаний для студентов:

- что уже реализовано и какими технологиями;
- какие инженерные практики применяются;
- как анализируется и тестируется код;
- какой реальный уровень подготовки можно считать достигнутым.

Документ основан на текущем состоянии репозитория LaravelExample.

## 1) Технологический стек и окружение

### Backend

- PHP: `^8.2`
- Laravel: `^12.0`
- Архитектурный стиль: Laravel MVC + сервисный слой + репозитории (гибридный учебный вариант)
- API: REST, версия `v1`, ресурсные роуты для `payments` и `accounts`

### Инфраструктура и запуск

- Контейнеризация: `docker-compose`
- Сервисы:
  - `php` (Apache + PHP 8.2, расширения `pdo_mysql`, `pcntl`, `bcmath`, `redis`, `xdebug`)
  - `queue-worker` (Supervisor)
  - `mysql:8.0`
  - `redis:7-alpine`
  - `adminer`
- Типичный сценарий запуска/проверки в проекте: через `docker compose ...`

### Frontend/tooling (базовый слой Laravel)

- Vite `^7`
- TailwindCSS `^4`
- axios, concurrently
- Это вспомогательный слой, основной фокус проекта - backend/API.

## 2) Ключевые пакеты Composer и их роль

### Production-зависимости

- `laravel/framework` - основной фреймворк, DI-container, HTTP lifecycle, Eloquent, queue/events.
- `laravel/sanctum` - инфраструктура токен-авторизации для API.
- `laravel/horizon` - мониторинг и управление очередями Redis.
- `sentry/sentry-laravel` - error tracking, контекст ошибок, tags/extra, трассировка.
- `spatie/laravel-data` - typed data-объекты/валидация (в проекте применено частично).
- `darkaonline/l5-swagger` - генерация/публикация OpenAPI-документации.
- `laravel/tinker` - интерактивная отладка/исследование моделей.

### Dev-зависимости

- `phpunit/phpunit` - unit/feature тестирование.
- `mockery/mockery` - мокирование контрактов в тестах.
- `phpstan/phpstan` - статический анализ (`level: 3`, анализ директории `app`).
- `friendsofphp/php-cs-fixer` + `laravel/pint` - форматирование и стиль.
- `fakerphp/faker` - тестовые/фейковые данные.
- `laravel/pail` - чтение логов во время разработки.
- `laravel/sail` - стандартный Laravel tooling для docker-окружения (дополняет текущий `docker-compose`).
- `nunomaduro/collision` - developer-friendly вывод ошибок в консоли.

## 3) Архитектура проекта (фактическая)

### 3.1 Слои обработки в API

Для `v1` используется устойчивый паттерн:

1. `routes/api.php` -> ресурсные маршруты.
2. Controller (`app/Http/Controllers/Api/V1/*Controller.php`) принимает HTTP.
3. FormRequest валидирует вход.
4. DTO собирает нормализованные данные.
5. Service реализует бизнес-логику.
6. Repository абстрагирует доступ к БД.
7. Resource формирует JSON-ответ.

Этот же подход повторяется для `payments` и `accounts`.

### 3.2 DI и контракты

- Используются интерфейсы репозиториев (`Contracts/Repositories/*Interface.php`) и биндинги в `AppServiceProvider`.
- В режиме `testing` интерфейсы подменяются на in-memory/mock реализации (`tests/Mocks/*`), в остальных средах - на реальные Eloquent-репозитории.
- Это сильная учебная практика: изоляция бизнес-логики от инфраструктуры и удобное тестирование.

### 3.3 Доменный фокус

Проект моделирует финансовый модуль:

- сущности: `Payment`, `Account`, `AuditLog`;
- ключевые операции: создание платежа, списание баланса, комиссии, аудит, уведомления;
- контекст продакшн-практик: ошибки платежей, correlation ID, observability.

### 3.4 Асинхронность и события

- Есть `Event` + `Listeners`, `Jobs`, `Notifications`.
- Есть заготовки для надежной обработки (retry/backoff/timeout в job-классах).
- Horizon присутствует как инфраструктура наблюдаемости очередей.
- В проекте это скорее учебно-демонстрационный слой зрелой архитектуры, чем полностью завершенный production pipeline.

### 3.5 Наблюдаемость (observability)

- Middleware `AssignCorrelationId`:
  - добавляет/пробрасывает `X-Correlation-ID`;
  - кладет request-контекст в `Context` (`correlation_id`, endpoint, user_id и т.д.);
  - шарит контекст в логи;
  - добавляет контекст в Sentry scope.
- Сервис `PaymentErrorReporter`:
  - структурированный `Log::error`;
  - дополнительно объединяет доменный контекст ошибки с `Context::all()`;
  - Sentry capture c tags/extra/user-контекстом.
- Кастомный exception rendering в `bootstrap/app.php` для API v1 (422/404/доменные ошибки).

### 3.6 Учебные зоны "плохой/хорошей архитектуры"

В проекте специально сохранены:

- `AntiPattern` (пример ухудшенной архитектуры: неименованные классы, смешение уровней, слабые контракты);
- `GoodPattern`/аккуратные сервисы (пример эволюции к более чистой структуре).

Это важно для обучения: можно показывать "как было" -> "как стало".

## 4) Данные, миграции и persistence

- База по умолчанию в runtime-инфраструктуре: MySQL (`docker-compose`).
- В тестах: SQLite in-memory (`phpunit.xml`).
- Миграции покрывают:
  - платежи;
  - аудит;
  - аккаунты;
  - jobs/cache/users/tokens.
- Persistence-паттерн: Eloquent + репозитории поверх Eloquent.

Это компромисс между "быстро в Laravel" и "архитектурно объяснимо студентам".

## 5) Как в проекте анализируется код

### Статика и стиль

- PHPStan (`composer stan`) анализирует `app` на уровне `3`.
- PHP CS Fixer:
  - `composer cs-check` для dry-run проверки;
  - `composer cs-fix` для автоисправления.
- Правила стиля:
  - PSR-12;
  - `declare(strict_types=1)`;
  - строгие параметры (`strict_param`).

### Архитектурный review-подход

В учебном процессе акцент на проверке:

- соблюдения слоев (Controller не должен содержать доменную логику);
- стабильности API-контрактов;
- корректности обработки ошибок и логирования;
- минимизации регрессий (через тесты и статический анализ).

## 6) Как в проекте тестируется код

### Test pyramid (фактически)

- Unit тесты: отдельные сервисы/калькуляторы.
- Feature/API тесты:
  - проверка endpoint-ов `/api/v1/*`;
  - статусы, JSON-структура, валидация, 404.
- Service-тесты с моками интерфейсов и с test-mocks через DI.

### Инфраструктура тестирования

- PHPUnit 11.
- База в тестах: SQLite `:memory:`.
- Часто используется `RefreshDatabase`.
- Есть Event::fake для изоляции побочных эффектов.
- Есть артефакты code coverage (`coverage-report`), т.е. практика покрытия уже внедрялась.

### Что это дает студентам

- понимание разницы unit vs feature;
- умение тестировать контракты API как "черный ящик";
- практику проверки бизнес-логики через моки и контракты.

## 7) API и документация контракта

- API v1 строится через `apiResource`.
- Используются FormRequest и Resource для стабильного входа/выхода.
- Подключен `l5-swagger` и есть `public/openapi.yaml`.
- В учебных материалах явно заложена связка:
  - OpenAPI/Swagger как контракт;
  - Feature tests как автоматическая верификация контракта.

## 8) Карта зрелости студентов по этому проекту

Ниже оценка "чему уже научили" исходя из кода и материалов модулей 7.3-8.2.

### Уже точно умеют

- строить CRUD API на Laravel;
- вводить валидацию через FormRequest;
- отделять HTTP слой от бизнес-логики через Service;
- использовать DTO/Value Object для доменных данных;
- работать с Eloquent и миграциями;
- писать Feature и Unit тесты;
- использовать DI и подмену реализаций в тестах;
- документировать API через OpenAPI/Swagger;
- внедрять базовый monitoring/error tracking (Sentry, correlation ID, structured logs);
- понимать базовый incident flow (алерт -> анализ -> fix/rollback).

### Это соответствует уровню

- уверенный Junior+ / Strong Junior backend (Laravel/PHP);
- частично Middle-практики уже есть:
  - контракты и слои;
  - observability;
  - тестирование и quality gates;
  - асинхронные механизмы (jobs/events).

### Что нужно для устойчивого Middle

- выровнять архитектуру (убрать/изолировать legacy-антипаттерны из runtime-потока);
- довести типизацию до конца (строгие сигнатуры интерфейсов/DTO без mixed/безтиповых параметров);
- усилить статический анализ (поднимать уровень PHPStan и закрывать baseline issues);
- довести event/listener/job pipeline до консистентного production-ready состояния;
- расширить интеграционные тесты (очереди, события, уведомления, error-flow);
- добавить явные NFR-практики: идемпотентность, retry policy с бизнес-ограничениями, SLO/alerting.

## 9) Практические правила для AI-агентов в этом репозитории

1. Предпочитать минимальный дифф и текущий паттерн `Controller -> Request -> DTO -> Service -> Repository`.
2. Не ломать публичный API-контракт (`/api/v1`, структура `data/errors`) без явного запроса.
3. Для новых ошибок использовать проектный observability-подход:
   - структурированные логи;
   - `correlation_id`;
   - Sentry контекст только с безопасными полями.
4. Для изменений в логике платежей обязательно добавлять/обновлять тесты.
5. Перед merge прогонять минимум:
   - `composer cs-check`
   - `composer stan`
   - `php artisan test`
6. Команды в первую очередь предполагаются внутри контейнера (`docker compose exec ...`) для一致ного окружения.
