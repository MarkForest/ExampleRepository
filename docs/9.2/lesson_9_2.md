# **Модуль 9. Основи продуктивності та оптимізації**

## **Урок 2. Кешування та оптимізація роботи з базою даних**



У **Уроці 9.1** ви з’ясували, де зазвичай виникають вузькі місця (БД, відсутність кешу, N+1), навіщо потрібні вимірювання та профайлінг, і який цикл «вимірюй → аналізуй → оптимізуй» застосовувати. Тепер переходимо до **практики**: як застосовувати кешування усвідомлено, як знаходити та усувати проблеми продуктивності в запитах до БД, як боротися з N+1 за допомогою eager loading та як використовувати індекси. Усе це в контексті **фінансового** застосунку: кеш довідників і налаштувань, списки платежів та рахунків, пагінація, jobs для важких звітів. Код у прикладах - **строго типізований**. Проєкт, як і раніше, розгортається через **docker-compose** (Redis для кешу, як у Модулі 5). Після цього уроку ви будете готові до Уроку 9.3: швидкість API на рівні HTTP, HTTP-кешування та підготовка до highload.



## **1. Мета уроку**

Мета цього уроку - навчити застосовувати **кешування та оптимізацію БД** усвідомлено, на основі аналізу, а не хаотично.

Після уроку ви повинні:

* розуміти різні типи кешування (дані, результати запитів, відповіді API) та коли який використовувати;
* знати, які дані варто кешувати, а які ні, та як інвалідувати кеш;
* вміти аналізувати запити до БД (профайлінг, кількість запитів);
* вміти усувати N+1 за допомогою eager loading (Laravel / Symfony);
* розуміти роль індексів та баланс індексації;
* бачити зв’язок між БД, кешем і швидкістю API.



## **2. Кешування - не «чарівна кнопка»**

Типова помилка: «Система повільна - давайте додамо кеш». Реальність: **неправильне кешування** призводить до складних багів (застарілі дані, некоректні баланси або списки); кеш **не виправляє погану архітектуру** (наприклад, N+1 лише приховується тимчасово, поки кеш теплий); **інвалідація кешу** - найскладніша частина: коли саме очищати або оновлювати кеш при зміні даних. Кешування - це **інструмент**, який потрібно застосовувати там, де є виміряна проблема та чіткі правила оновлення. У фінансовому застосунку особливо небезпечно кешувати дані, що впливають на грошові операції (баланс, ліміти), без продуманої інвалідації.



## **3. Типи кешування у веб-застосунках**

На високому рівні кешують: **дані** - довідники, налаштування, рідко змінювані сутності (валюти, типи операцій, комісії); **результати запитів** - важкі SQL-агрегації, складні фільтри, списки з обчисленнями; **об’єкти** - серіалізовані моделі або DTO; **відповіді API** - цілі JSON-відповіді для read-only або рідко змінюваних ендпоінтів (наприклад, умови тарифів, публічний каталог). Кожен тип має свої особливості: для даних важлива інвалідація при оновленні; для результатів запитів - ключ за параметрами (account_id, період) та TTL; для API-відповіді - умови кешування (Cache-Control, ETag) та те, що не всі ендпоінти можна кешувати (персоналізовані, мутаційні).



## **4. Кешування даних (довідники, налаштування)**

Кешування даних зменшує кількість запитів до БД для рідко змінюваних сутностей. Приклади у фінансовому контексті: список валют; типи операцій (поповнення, списання, переказ); налаштування комісій або лімітів (якщо змінюються рідко); коди відповіді платіжного шлюзу (довідник для відображення помилок). Правила: кешувати **результат** (наприклад, масив або колекцію), а не «запит»; задати **TTL** (час життя) або інвалідувати при оновленні цих даних в адмінці; мати **fallback** - якщо кеш порожній або Redis недоступний, брати з БД. У Laravel: `Cache::remember('currencies', 3600, fn () => Currency::all());`; при оновленні довідника - `Cache::forget('currencies')` або тег. У Symfony - аналогічно через `CacheInterface::get()` та callback.



## **5. Кешування результатів запитів**

Кешуються результати **важких** запитів: агрегації (сума платежів за період по рахунках), складні фільтри з JOIN, списки з сортуванням по великих таблицях. Важливо: кешувати саме **результат** з урахуванням параметрів (ключ кешу має включати account_id, date_from, date_to тощо); задати **TTL** або інвалідацію при появі нових платежів по цьому рахунку; не кешувати реальний баланс у реальному часі без чіткої політики - інакше ризик показувати застарілі суми. Приклад ключа: `payments_summary:account:123:2025-02`. Інвалідація: при створенні/оновленні платежу для account_id=123 очищати або оновлювати відповідні ключі кешу для цього рахунку.



## **6. Кешування відповідей API**

API-кеш зменшує навантаження на застосунок і БД: клієнт або проксі повертає збережену відповідь без виконання бізнес-логіки. Підходить для: **GET**-запитів; read-only ресурсів; публічних або рідко змінюваних даних (тарифи, умови, каталог). **Не підходить** для: персоналізованих даних без спеціальної логіки (наприклад, «мої платежі» - залежить від user_id); POST/PATCH/DELETE; даних у реальному часі (поточний баланс для критичних операцій). У Уроці 9.3 ми розглянемо HTTP-кешування (Cache-Control, ETag) на рівні протоколу. На рівні застосунку можна кешувати відповідь у Redis за ключем (наприклад, `api:v1:tariffs`) з TTL та інвалідацією при зміні даних.



## **7. Коли кешування НЕ підходить**

Не варто кешувати: **часто змінювані дані** без короткого TTL та чіткої інвалідації (щойно створені платежі в «останні операції» можна кешувати на 10–30 секунд, але не годину); **персоналізовані критичні відповіді** (баланс для підтвердження платежу має бути актуальним); **процеси з жорсткою консистентністю** (наприклад, перевірка ліміту перед списанням). Якщо кешувати такі речі, потрібні дуже короткий TTL або негайна інвалідація при кожній зміні - інакше ризик фінансових помилок та плутанини для користувача.



## **8. Інвалідація кешу**

Класичне питання: «Коли очищати кеш?» Стратегії: **TTL (час життя)** - кеш живє N секунд/хвилин, після чого застарілий (простій підхід для довідників); **ручна інвалідація** - при оновленні даних у коді викликати `Cache::forget($key)` або очищати за тегом (наприклад, при оновленні комісій - очистити `settings:commission`); **події (events)** - при збереженні платежу слухач очищає кеш «підсумки по рахунку» для цього account_id; **комбіновані підходи** - TTL + інвалідація при записі. Інвалідація - найскладніша частина кешування: занадто довгий TTL дає застарілі дані, занадто агресивна інвалідація зменшує виграш від кешу. У фінансовому застосунку для балансів та критичних сум краще не покладатися на довгий кеш без чіткої політики.



## **9. Redis як кеш (концептуально)**

Redis - in-memory сховище ключ-значення, дуже швидкий; часто використовується як кеш-шар у веб-застосунках. Підтримує TTL для ключів, що зручно для тимчасового кешу. У Laravel Redis - стандартний драйвер кешу (`CACHE_DRIVER=redis`); у Symfony - через `cache.adapter.redis`. У **docker-compose** Redis вже може бути присутній для черг (Модуль 5); той самий інстанс часто використовують і для кешу. Redis - **не база даних**: не зберігайте єдину копію критичних даних тільки в Redis; кеш може бути очищений або втрачений, застосунок має коректно працювати при fallback на БД.



## **10. Ризики неправильного використання Redis**

Типові помилки: зберігання **занадто великих** об’ємів (великі JSON) - зайва пам’ять та повільна серіалізація; **відсутність TTL** - ключі накопичуються, пам’ять закінчується; кешування **всього підряд** без аналізу - складна інвалідація та баги; **відсутність fallback** - при недоступності Redis застосунок падає замість того, щоб брати дані з БД. Рекомендації: задавати TTL для більшості ключів; обмежувати розмір значень; обробляти винятки кешу та переходити на БД при помилці Redis.



## **11. Профайлінг запитів до БД**

Перш ніж оптимізувати запити, потрібно **побачити їх**: скільки запитів виконується на один HTTP-запит; який час кожного; чи є повторювані або дубльовані запити. Інструменти: **Laravel Telescope** - вкладка Queries з списком та часом; **Laravel Debugbar** - панель з запитами; **логування SQL** (у development) - обережно, не засмічувати продакшн. У Symfony - Doctrine Profiler, Blackfire або аналогічні інструменти. Мета - знайти N+1 (багато однакових запитів з різними id), один дуже довгий запит або зайві запити в циклі.



## **12. Проблема N+1 (детальніше)**

Сценарій: отримуємо список платежів (один запит); для кожного платежу у відповіді потрібні рахунок та користувач; якщо зв’язки завантажуються **lazy** (при зверненні до `$payment->account`), для кожного платежу виконується окремий запит. Результат: 1 запит для списку + N запитів для зв’язок = N+1 запитів. При 50 платежах - 51 запит; при 500 - 501. Наслідок - деградація продуктивності та ризик таймаутів. Рішення - **eager loading**: завантажити зв’язки **заздалегідь** одним або кількома запитами. У Laravel: `Payment::with(['account', 'user'])->where(...)->paginate()`; у Doctrine: join або fetch у запиті. Lazy loading у списках - майже завжди проблема; у деталях одного запису він прийнятний.



## **13. Eager loading у Laravel**

У Laravel для зв’язок Eloquent використовується метод **`with()`**: він додає додаткові запити для завантаження зв’язаних моделей одним пакетом, замість запиту на кожен елемент колекції. Приклад: замість `Payment::where('account_id', $id)->get()` і потім у циклі `$payment->account`, використовуємо `Payment::with('account')->where('account_id', $id)->get()`. Тоді виконується два запити: один для платежів, один для всіх рахунків, що потрібні (або один запит з join, залежно від зв’язку). Для вкладених зв’язок: `with(['account', 'account.user'])`. У наступному розділі - приклад коду з типізацією для репозиторія платежів.



## **14. Eager loading у Symfony / Doctrine**

У Doctrine зв’язки за замовчуванням завантажуються lazy (при зверненні до властивості). Для списків потрібно завантажити зв’язки **явно**: через DQL join (`SELECT p FROM Payment p JOIN p.account a WHERE p.account = :account`) або через `QueryBuilder` з join; або через Doctrine `EntityRepository` з методом, що робить join і повертає масив/колекцію. Тоді в одному або кількох запитах отримуємо платежі разом з рахунками (і при потребі користувачами), без N+1. Принцип той самий, що й у Laravel: не покладатися на lazy loading у циклі по колекції.



## **15. Індекси: що важливо розуміти**

Індекси прискорюють **читання** (WHERE, JOIN, ORDER BY) і уповільнюють **запис** (INSERT/UPDATE мають оновлювати індекс); вони займають місце на диску. Важливо: **індексувати поля пошуку та фільтрації** (account_id, created_at, status для платежів); **індексувати foreign keys**, по яких часто роблять join; **не індексувати «про всяк випадок»** - кожен індекс має відповідати конкретному запиту або типу запитів. Як зрозуміти, що потрібен індекс: повільні WHERE/JOIN по полю; великі таблиці; часті фільтри по одному й тому ж полю; використання ORDER BY по полю. У міграціях Laravel/Doctrine додають індекси для таких полів; аналіз повільних запитів (EXPLAIN) допомагає перевірити використання індексів.



## **16. Пагінація як must-have для списків**

API без пагінації для списків (платежі, рахунки, транзакції) **не масштабується**: при зрості даних відповідь стає величезною, час виконання та пам’ять зростають, клієнт може отримати таймаут. Пагінація обмежує розмір відповіді (наприклад, 20–50 записів на сторінку), стабілізує час відповіді та навантаження на БД. У Laravel - `Model::paginate($perPage)`; у Symfony - Doctrine Paginator або аналог. Пагінація - не опція для будь-якого ендпоінта, що повертає список з потенційно великої таблиці; це **must-have** для production-ready API.



## **17. Jobs як інструмент оптимізації**

Важку логіку (генерація звіту, експорт виписки, масовий підрахунок) не варто виконувати синхронно в HTTP-запиті: це блокує відповідь та навантажує веб-процес. **Jobs** (черги - Модуль 5) дозволяють виконувати такі задачі асинхронно: API приймає запит, ставить завдання в чергу, швидко повертає «завдання прийнято» або посилання на результат; воркер виконує обчислення у фоні. Таким чином API залишається швидким, а важкі операції не блокують користувача. У фінансовому контексті типовий приклад - експорт виписки за період або генерація звіту по рахунках: запит створює job і повертає 202 Accepted з ідентифікатором завдання; клієнт отримує результат по окремому ендпоінту або через нотифікацію.



## **18. Приклад коду: репозиторій з eager loading та пагінацією**

Нижче - фрагмент репозиторія для списку платежів рахунку з eager loading та пагінацією. Код строго типізований; у фінансовому контексті повертаємо лише потрібні зв’язки (рахунок, користувач для відображення), без зайвих даних.

**Laravel - репозиторій платежів з eager loading:**

```php
<?php

declare(strict_types=1);

namespace App\Repositories;

use App\Models\Payment;
use Illuminate\Contracts\Pagination\LengthAwarePaginator;
use Illuminate\Database\Eloquent\Collection;

final class PaymentRepository
{
    /**
     * Список платежів рахунку з рахунком та ініціатором (eager loading, пагінація).
     *
     * @return LengthAwarePaginator<Payment>
     */
    public function getByAccountIdPaginated(int $accountId, int $perPage = 20): LengthAwarePaginator
    {
        return Payment::query()
            ->with(['account', 'user'])
            ->where('account_id', $accountId)
            ->orderByDesc('created_at')
            ->paginate($perPage);
    }

    /**
     * Останні платежі для дашборду (обмежений набір, можна кешувати на 30 сек).
     *
     * @return Collection<int, Payment>
     */
    public function getLatestByAccountId(int $accountId, int $limit = 10): Collection
    {
        return Payment::query()
            ->with(['account:id,name', 'user:id,email'])
            ->where('account_id', $accountId)
            ->orderByDesc('created_at')
            ->limit($limit)
            ->get();
    }
}
```

**Symfony / Doctrine - аналог з join:**

```php
<?php

declare(strict_types=1);

namespace App\Repository;

use App\Entity\Payment;
use Doctrine\Bundle\DoctrineBundle\Repository\ServiceEntityRepository;
use Doctrine\ORM\Tools\Pagination\Paginator;
use Doctrine\Persistence\ManagerRegistry;

/**
 * @extends ServiceEntityRepository<Payment>
 */
final class PaymentRepository extends ServiceEntityRepository
{
    public function __construct(ManagerRegistry $registry)
    {
        parent::__construct($registry, Payment::class);
    }

    /**
     * Список платежів рахунку з eager load account та user (один запит з join).
     *
     * @return Paginator<Payment>
     */
    public function getByAccountIdPaginated(int $accountId, int $page = 1, int $perPage = 20): Paginator
    {
        $qb = $this->createQueryBuilder('p')
            ->innerJoin('p.account', 'a')->addSelect('a')
            ->leftJoin('p.user', 'u')->addSelect('u')
            ->where('p.account = :accountId')
            ->setParameter('accountId', $accountId)
            ->orderBy('p.createdAt', 'DESC');

        $query = $qb->getQuery()
            ->setFirstResult(($page - 1) * $perPage)
            ->setMaxResults($perPage);

        return new Paginator($query);
    }
}
```

У обох прикладах зв’язки завантажуються одним запитом (або мінімальною кількістю), а не в циклі - N+1 усунуто. Пагінація обмежує об’єм даних.



## **19. Приклад коду: кеш довідника з інвалідацією**

**Laravel - кеш списку валют з TTL та можливістю інвалідації:**

```php
<?php

declare(strict_types=1);

namespace App\Services;

use App\Models\Currency;
use Illuminate\Support\Collection;
use Illuminate\Support\Facades\Cache;

final class CurrencyService
{
    private const CACHE_KEY = 'currencies:list';
    private const TTL_SECONDS = 3600;

    /**
     * Список валют (кешований). Інвалідувати при оновленні довідника.
     *
     * @return Collection<int, Currency>
     */
    public function getAll(): Collection
    {
        return Cache::remember(self::CACHE_KEY, self::TTL_SECONDS, function (): Collection {
            return Currency::query()->orderBy('code')->get();
        });
    }

    public function invalidateCache(): void
    {
        Cache::forget(self::CACHE_KEY);
    }
}
```

При оновленні валют у адмінці викликати `$currencyService->invalidateCache()`, щоб наступний запит отримав свіжі дані з БД і знову закешував їх.



## **20. Типові помилки при оптимізації**

Уникайте: **кешування без аналізу** - кешувати все підряд без вимірювань та правил інвалідації; **ігнорування N+1** - не перевіряти кількість запитів у профайлері; **відсутності індексів** - повільні фільтри по великих таблицях без індексів; **передчасної оптимізації** - складні кеш-шари до виявлення реального вузького місця; **оптимізації не там** - полірувати PHP замість одного повільного SQL. Системний підхід: профайлінг → виявлення N+1 або повільного запиту → eager loading / індекс / кеш → перевірка результату.



## **21. Практична частина уроку**

Усі завдання - у **фінансовому** контексті.

### **Завдання на уроці**

1. **Список ендпоінтів і пагінація/кеш**  
   Виберіть список ендпоінтів у проєкті (наприклад, платежі, рахунки, виписки). Для кожного визначте: чи є пагінація; чи потрібна; чи можливе кешування (read-only, рідко змінювані дані) і з яким TTL або умовами інвалідації.

2. **Аналіз одного ендпоінту**  
   Виберіть один ендпоінт списку (наприклад, GET платежів по рахунку). За допомогою Telescope або Debugbar проаналізуйте: скільки запитів до БД виконується; чи є N+1 (повторювані запити за зв’язками); чи є зайві запити. Опишіть, що потрібно змінити (eager loading, обмеження полів).

3. **Eager loading та індекси**  
   Запропонуйте для цього ендпоінту: які зв’язки завантажувати через eager loading (назви методів/зв’язків); які індекси перевірити або додати (поля для WHERE, ORDER BY, foreign keys). Якщо можливо - реалізуйте зміни в репозиторії (типізований код) та перевірте кількість запитів до і після.

4. **Кеш і інвалідація**  
   Виберіть один тип даних для кешування (наприклад, довідник валют або налаштування комісій). Опишіть: ключ кешу, TTL, коли інвалідувати (при якій події або дії). Напишіть фрагмент коду (Laravel або Symfony) з типізацією: отримання з кешу з fallback на БД та метод інвалідації.

5. **Компроміси**  
   Опишіть компроміси: між TTL кешу та актуальністю даних; між кількістю індексів та швидкістю запису; між синхронною відповіддю та винесенням у job (швидка відповідь vs негайний результат).
