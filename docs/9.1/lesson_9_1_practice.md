# Практичні приклади до уроку 9.1 - Продуктивність як системне мислення: де виникають проблеми і з чого починати

> **Загальна ідея практики:**  
> Навчитися **дивитися на продуктивність системно**: не оптимізувати «в темну», а аналізувати конкретний API‑ендпоінт (наприклад, список платежів), виділяти потенційні вузькі місця (БД, N+1, кеш, об’єм даних) і формулювати **перший усвідомлений крок оптимізації**.  
> Ці приклади не привʼязані до конкретного фреймворку, але легко вкладаються в архітектуру Laravel/Symfony, яку ви будували в попередніх модулях.



## 1. Блок 3: Аналіз API‑ендпоінта → Слайд 8a

**Загальна ідея прикладу:**  
Проаналізувати один конкретний ендпоінт у фінансовому домені, наприклад:  
`GET /api/v1/accounts/{id}/payments` - список платежів рахунку.  
Мета - **описати потік даних** і виявити місця, де потенційно «боляче».

### 1.1. Умовний ендпоінт

```http
GET /api/v1/accounts/{accountId}/payments?page=1&per_page=50
Accept: application/json
```

### 1.2. Потік даних (концептуально)

1. Контролер отримує `accountId` та параметри пагінації.
2. Сервіс перевіряє, що рахунок належить поточному користувачу.
3. Сервіс звертається до репозиторію:
   - `PaymentRepository::findByAccountWithFilters($accountId, $filters, $pagination)`.
4. Репозиторій виконує один або кілька SQL‑запитів:
   - основний SELECT по таблиці `payments`;
   - при поганій реалізації - N+1 до `accounts` або `users`.
5. Результат перетворюється в DTO/Resource і повертається як JSON.

### 1.3. Питання для аналізу

- Скільки запитів до БД виконується?
- Чи є пагінація (і реальна ли она реализована в SQL, а не «обрезкой массива»)?
- Чи выбираются только нужные поля или `SELECT *`?
- Чи есть N+1 (дополнительные запросы по связям)?
- Чи используется какой‑то кеш (справочники, фильтры, результаты агрегаций)?
- Чи не выполняется тяжелая бизнес‑логика в цикле (для каждого платежа)?

> **Що зробити на практиці:**  
> Взять свой реальный endpoint (не обязательно именно этот) и по аналогичным пунктам описать его поведение: от HTTP до БД и обратно. Это база для следующих шагов.



## 2. Блок 4: Кеш і не‑кеш → Слайд 11a

**Загальна ідея прикладу:**  
На том же эндпоинте понять, **что можно кешировать, а что лучше не трогать**, и почему.

### 2.1. Что можно кешировать в `GET /accounts/{id}/payments`

- **Справочники**, участвующие в фильтрации/отображении:
  - список типов операций (пополнение, списание, перевод);
  - статусы платежей (processed, pending, failed).
- **Агрегированные данные**:
  - общая сумма платежей за период (для дашборда);
  - количество платежей по статусам.
- **Read‑only блоки UI**:
  - настройки фильтров/предустановленные диапазоны дат;
  - публичные тарифы/комиссии (если меняются редко).

### 2.2. Что НЕ стоит кешировать (или очень аккуратно)

- **Текущий баланс счета** - должен быть максимально актуальным.
- **Только что созданные платежи** в критичных сценариях (пока пользователь ожидает подтверждения).
- **Персонализированные данные с жесткими требованиями по консистентности**, где ошибка из‑за старого кеша неприемлема.

### 2.3. Пример идеи кеша (на уровне концепта)

> Кешируем список **типов операций** и **статусов платежей** на 10–30 минут в Redis.  
> При отрисовке фильтров и отображении платежей не ходим каждый раз в БД за справочниками.

> **Що зробити на практиці:**  
> - выписать для своего endpoint’а или модуля:  
>   - 2–3 кандидата на кеш;  
>   - 2–3 вещи, которые кешировать пока нельзя (и почему).



## 3. Блок 5: N+1 → Слайд 14a

**Загальна ідея прикладу:**  
Показать, как **концептуально** возникает N+1 в финансовом модуле и как это масштабируется.

### 3.1. Пример потенциального N+1

```php
// Псевдокод
$payments = Payment::query()
    ->where('account_id', $accountId)
    ->orderByDesc('created_at')
    ->take(50)
    ->get();

foreach ($payments as $payment) {
    // Для каждого платежа отдельный запрос за счетом
    $account = $payment->account; // ленивый загруз
    // И/или за пользователем
    $user = $payment->user;
}
```

- При 50 платежах → 1 (список) + 50 (счета) + 50 (пользователи) = **101 запрос**.
- При 500 платежах (без пагинации) - 1 + 500 + 500 = **1001 запрос**.

### 3.2. Как это влияет при росте данных

- Время ответа растет примерно линейно с количеством записей.
- Нагрузка на БД увеличивается многократно.
- При десятках тысяч записей легко получить **таймаут**.

> **Що зробити на практиці:**  
> - найти у себя места, где:  
>   - вы получаете коллекцию сущностей (платежи, счета, пользователи);  
>   - внутри цикла обращаетесь к связям, которые подгружаются лениво;  
> - прикинуть, во сколько запросов это выливается при 100/1000/10000 элементах.



## 4. Блок 6: Перший крок оптимізації → Слайд 17a

**Загальна ідея прикладу:**  
Не пытаться «оптимизировать всё», а выбрать **один** понятный первый шаг, который даёт максимальный эффект и легко измерим.

### 4.1. Примеры первых шагов

Для `GET /accounts/{id}/payments`:

- **Включить eager loading** связей:

```php
$payments = Payment::query()
    ->where('account_id', $accountId)
    ->with(['account', 'user']) // или только то, что реально нужно
    ->orderByDesc('created_at')
    ->paginate(50);
```

- **Ввести пагинацию**, если её нет:
  - вместо `get()` использовать `paginate()`/`cursorPaginate()`;
  - ограничить `per_page` (например, максимум 100).

- **Проверить индексы**:
  - убедиться, что есть индекс по `account_id` и, возможно, составной по (`account_id`, `created_at`).

- **Урезать ответ**:
  - через DTO/Resource отдавать только нужные поля, без тяжёлых вложенных объектов.

### 4.2. Как измерять эффект

- До изменения:
  - посмотреть в Telescope/Debugbar/логах:  
    - время запроса;  
    - количество SQL‑запросов;  
    - «тяжёлые» запросы.
- После изменения:
  - повторить измерения;  
  - сравнить время ответа и количество запросов.

> **Що зробити на практиці:**  
> - выбрать **одну** оптимизацию для своего endpoint’а (eager loading, пагинация, первичный кеш, индекс);  
> - описать, **как именно** вы будете мерить улучшение (время, количество запросов, нагрузка на БД).



## 5. Підсумок практики

> - Оптимизация начинается не с кода, а с **анализа конкретного сценария** (endpoint, поток данных).  
> - **Кеширование** - мощный, но опасный инструмент: начинайте с справочников и агрегатов, а не с критичных real‑time данных.  
> - **N+1** и отсутствие пагинации - самые частые «легкоисправимые» проблемы.  
> - **Первый шаг оптимизации** должен быть маленьким, измеримым и бить в самое узкое место (по данным профайлера/логов), а не в случайный участок кода.

Ця практика готує ґрунт для наступних уроків Модуля 9, де ви вже побачите конкретні SQL‑патерни, кеш‑стратегії та профайлінг на прикладах коду.
