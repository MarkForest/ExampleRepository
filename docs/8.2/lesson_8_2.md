# **Модуль 8. Моніторинг та логування**

## **Урок 2. Error Tracking та аналіз помилок: Sentry і продакшн-практики**



У **Уроці 8.1** ви зрозуміли різницю між логуванням і моніторингом, навіщо потрібні рівні логування та контекст, що таке error tracking і як він доповнює логи. Тепер переходимо до **практики**: як інтегрувати **Sentry** у Laravel-застосунок, які помилки відправляти, який контекст додавати, як аналізувати помилки та пріоритизувати виправлення. Усе це розглядаємо в контексті **фінансового** застосунку: помилки платежів, інтеграції з платіжними шлюзами, бізнес-винятки та системні збої. Код у прикладах - **строго типізований**. Після цього уроку ви будете готові до Уроку 8.3 (лог-агрегація, структуровані логи, повний цикл інцидентів). Проєкт, як і раніше, може працювати в **docker-compose**; Sentry отримує події з застосунку по мережі, тому достатньо налаштувати DSN у `.env`.



## **1. Мета уроку**

Мета цього уроку - навчити **контролювати помилки в реальному середовищі** за допомогою error tracking.

Після уроку ви повинні:

* розуміти, як працює error tracking (збір, групування, контекст, сповіщення);
* вміти інтегрувати Sentry у Laravel (DSN, середовища, sampling);
* додавати контекст до помилок (user, payment_id, tags, extra);
* аналізувати помилки в Sentry: частота, affected users, регресії;
* визначати пріоритети виправлення (вплив на бізнес, кількість постраждалих);
* розуміти базовий процес роботи з інцидентом від алерту до виправлення.



## **2. Чому try/catch і логи - недостатньо**

Класичний підхід без error tracking: exception ловимо у глобальному Handler, логуємо в файл або stdout, користувач отримує 500; розробник дізнається про проблему випадково або від підтримки. Проблеми: немає **статистики** (скільки разів та сама помилка); немає **групування** (одна й та сама помилка в різних місцях зливається в один issue); немає **контексту** у зручному вигляді (user, request, tags); немає **сповіщень** при першій появі. Error tracking вирішує це системно: кожен exception потрапляє в один інтерфейс, групується, доповнюється контекстом і може тригерити алерт. У фінансовому застосунку це дозволяє реагувати на падіння платіжного модуля до масових скарг.



## **3. Що таке Error Tracking на практиці**

Error tracking - це сервіс, який: автоматично ловить необроблені (і за бажанням оброблені) exceptions; **групує** однакові помилки за типом і stack trace; показує **де, коли і чому** вони виникають; дозволяє відстежувати **частоту** та **affected users**; може **сповіщати** команду (email, Slack, PagerDuty). Це не «просто лог», а інструмент **управління якістю**: один баг = один issue з історією та контекстом, а не тисячі рядків у файлах.



## **4. Чому Sentry - популярний вибір**

Sentry легко інтегрується з Laravel (офіційний пакет `sentry/sentry-laravel`), підтримує контекст (user, request, tags, extra data), має зручний UI для перегляду та групування, показує регресії (з’явилася після релізу), інтегрується з релізами та комітами. Принципи роботи схожі в інших систем (Bugsnag, Rollbar, GlitchTip): головне - налаштувати відправку потрібних помилок та контексту; фінансовий контекст (payment_id, account_id, сума) критично важливий для швидкого розслідування.



## **5. Які помилки повинні потрапляти в Sentry**

**Повинні потрапляти:** неочікувані exceptions (uncaught) у будь-якому місці; критичні бізнес-помилки, які ви вирішили логувати в Sentry (наприклад, невдала інтеграція з платіжним шлюзом після кількох retry); помилки зовнішніх API (таймаути, 5xx від провайдера); помилки, що впливають на результат для користувача (платіж не створився через збій БД). **Не повинні:** очікувані валідаційні помилки (422 з форми); контрольовані бізнес-відмови, які повертаються клієнту як 400/422 (наприклад, «недостатньо коштів») - інакше Sentry переповниться «шумом». Виняток: якщо ви хочете аналізувати частоту таких відмов, можна відправляти їх окремим типом події з низьким пріоритетом або лише агреговано.



## **6. Інтеграція Sentry у Laravel**

Встановлення: `composer require sentry/sentry-laravel`. Після публікації конфігу та додавання **DSN** у `.env` (`SENTRY_LARAVEL_DSN=https://...@sentry.io/...`) необроблені exceptions автоматично відправляються в Sentry. Додатково налаштовують: **середовище** (`APP_ENV` передається в Sentry, щоб фільтрувати local/staging/production); **sampling** (відсоток подій для відправки при дуже високому навантаженні); **release** (версія/коміт для прив’язки помилок до релізу). У `config/sentry.php` можна виключити певні типи винятків (наприклад, 404 або ValidationException) з відправки. У **docker-compose** достатньо мати мережевий доступ з контейнера застосунку до Sentry; окремий сервіс для Sentry не потрібен (Sentry - хостований сервіс).



## **7. Середовища і error tracking**

Важливо розділяти середовища: **local**, **staging**, **production**. У Sentry помилки групуються та фільтруються по `environment`; за замовчуванням Laravel передає `APP_ENV`. Щоб не засмічувати продакшн-проєкт, з local часто **не відправляють** події (перевірити в конфігу: `'send_default_pii'` та умови для `environment`). У production і staging Sentry увімкнений; так команда бачить лише релевантні помилки і не плутає їх з тим, що виникає на машинах розробників.



## **8. Контекст - ключ до корисних помилок**

Помилка без контексту: «Undefined array key "amount"» - незрозуміло, який користувач, який платіж, який запит. Помилка з контекстом: user_id=123, payment_id=456, endpoint=POST /api/v1/payments, amount=100.00, correlation_id=abc-123 - можна одразу відтворити сценарій і перевірити дані. **Контекст перетворює помилку на інформацію.** У Sentry контекст додають через SDK: `\Sentry\configureScope()` або перед викиданням винятку через багатий контекст у Laravel (user автоматично підставляється, якщо увімкнено; додатково додають tags та extra).



## **9. Який контекст варто додавати**

Типовий корисний контекст для фінансового застосунку: **user_id** (або id анонімного користувача); **payment_id**, **order_id**, **account_id** - щоб одразу знайти операцію; **HTTP method і URL** (endpoint); **correlation_id** (request id) для зв’язку з логами; **env** (production/staging). Додатково: сума, валюта, код відповіді платіжного шлюзу - у **extra** або **tags**, щоб можна було фільтрувати в Sentry (наприклад, усі помилки з gateway_code=timeout). Це значно скорочує час розслідування: замість «де це сталося» ви одразу бачите, який платіж і який клієнт постраждали.



## **10. Tags та extra data**

У Sentry **tags** - це пари ключ-значення для **фільтрації** (наприклад, `module=payments`, `action=create`, `gateway=stripe`). Їх варто тримати короткими та не містити персональних даних; вони допомагають швидко знайти всі помилки платіжного модуля або всі помилки з певним шлюзом. **Extra** - довільні дані для **перегляду** в деталях події: payload запиту (обережно з PII), відповідь шлюзу, проміжні значення. У Laravel контекст можна додавати глобально в middleware (correlation_id, endpoint) - тоді він автоматично потрапляє в кожну подію того запиту; або локально в місці виникнення помилки через `\Sentry\withScope()` і `captureException()` - зручно для платіжного сервісу: перед викликом шлюзу встановити tags і extra для поточного платежу, щоб при будь-якому exception вони вже були в події. Нижче в прикладі коду показано, як це зробити з типізацією та без витоку конфіденційних даних.



## **11. Групування помилок**

Sentry групує помилки за **типом exception** та **stack trace** (місце в коді). Одна й та сама помилка в одному рядку коду з’являється як один **issue** з лічильником подій та списком останніх occurrence. Це дозволяє бачити **одну проблему**, а не тисячу окремих записів; оцінювати масштаб (скільки разів, скільки користувачів); визначати пріоритети (найчастіша або найкритичніша спочатку). Якщо одна й та сама помилка має різний контекст (наприклад, різні payment_id), вона все одно групується в один issue - контекст видно в кожному occurrence.



## **12. Аналіз помилки: як читати Sentry**

При аналізі варто: подивитися **частоту** (скільки подій за годину/день); **перший і останній** occurrence (чи це регресія після деплою); **affected users** (скільки унікальних користувачів); **контекст** (user_id, payment_id, tags, extra); **release** - після якого коміту з’явилася помилка. Stack trace - лише початок; контекст і тренди дають повну картину. У фінансовому контексті важливо перевірити, чи помилка стосується одного клієнта чи масової інтеграції (наприклад, збій шлюзу для всіх).



## **13. Регресії**

**Регресія** - помилка, яка з’явилася після певного деплою; раніше її не було. Sentry дозволяє прив’язувати події до **release** (версія, git commit); тоді видно, після якого релізу помилка почала з’являтися. Це допомагає швидко визначити, чи варто відкотити реліз, чи шукати причину в новому коді. Для фінансових систем регресії після оновлення платіжного модуля особливо критичні - їх потрібно виявляти і виправляти в пріоритетному порядку.



## **14. Пріоритизація помилок**

Не всі помилки однаково важливі. Критерії пріоритету: **кількість постраждалих користувачів**; **частота** (масова помилка vs одиничні випадки); **вплив на бізнес** (платежі не проходять важливіше, ніж помилка відображення профілю); **тип функціоналу** (критичні грошові операції vs допоміжні сторінки). Не потрібно «гасити все одразу» - спочатку вирішують те, що блокує платежі або масово впливає на користувачів. Sentry дозволяє сортувати issues за частотою та числом users; це допомагає вибирати порядок виправлень.



## **15. Error tracking і бізнес**

Помилки впливають на довіру користувачів, дохід, навантаження на підтримку та репутацію. Error tracking дає **дані** для аргументації пріоритетів: «за останню добу ця помилка вплинула на 50 платежів» звучить переконливіше, ніж «іноді щось падає». Це допомагає узгоджувати з бізнесом, що виправляти в першу чергу, і приймати рішення на основі фактів, а не інтуїції.



## **16. Інциденти і error tracking**

Інцидент часто починається з: різкого росту помилок у Sentry; алерту (інтеграція Sentry з Slack або PagerDuty); скарг користувачів на недоступність або невдалі платежі. Error tracking дає **сигнал** і **дані**: який exception, де в коді, скільки подій, які користувачі. Далі команда переходить до логів (correlation_id, повна хронологія) та виправлення. Без error tracking перший крок - «здогадки»; з Sentry - «бачу issue X, за останні 10 хвилин 100 подій, endpoint POST /api/v1/payments».



## **17. Basic Incident Management (практично)**

Типовий сценарій: (1) Алерт з Sentry або моніторингу - «багато помилок у payments». (2) Визначення масштабу - відкрити issue в Sentry, подивитися частоту та affected users. (3) Тимчасове рішення - якщо потрібно: feature flag, відкат релізу, повідомлення користувачам. (4) Виправлення - патч, деплой, перевірка. (5) Аналіз причин - post-mortem: чому сталося, як уникнути. Без error tracking перший крок замість «бачу конкретний exception» стає «хтось сказав, що щось не працює».



## **18. Типові помилки при використанні Sentry**

* Відправляти **всі** помилки без фільтра - валідація, 404, очікувані бізнес-відмови засмічують проєкт.
* Ігнорувати **контекст** - помилки приходять без payment_id/user_id, розслідування ускладнене.
* Не закривати **resolved** issues після виправлення - накопичується хаос.
* Не аналізувати **тренди** - не видно регресій та зростання проблем.
* Не пов’язувати помилки з **релізами** - незрозуміло, після якого деплою щось зламалося.



## **19. Взаємодія Sentry і логів**

Sentry показує **що зламалося** (exception, stack trace, affected users). Логи показують **як і чому** - повну хронологію запиту, бізнес-кроки, відповіді зовнішніх API. Для глибокого розслідування після того, як Sentry вказав на issue, команда йде в лог-агрегацію за correlation_id або payment_id і переглядає всі події того запиту. Це **два доповнюючі інструменти**: Sentry для виявлення та пріоритизації, логи - для детального аналізу.



## **20. Sentry у Symfony (альтернатива)**

У Symfony для error tracking використовують пакет **sentry/sentry-symfony**: `composer require sentry/sentry-symfony`. Конфігурація через `.env` (`SENTRY_DSN=...`) та `config/packages/sentry.yaml`; необроблені винятки автоматично перехоплюються через інтеграцію з Kernel. Як і в Laravel, налаштовують середовище (`APP_ENV`), release, sampling; виключення певних типів винятків (наприклад, `NotFoundHttpException`) - через конфіг або `BeforeSendCallback`. Контекст (user, tags, extra) додають через `\Sentry\configureScope()` або `\Sentry\withScope()` перед викликом, що викликає exception; принципи ті самі, що й у Laravel: для фінансового модуля обов’язково передавати payment_id, account_id, endpoint у tags/extra, щоб у Sentry одразу було видно контекст помилки.



## **21. Приклад коду: контекст Sentry у платіжному сервісі**

Нижче - фрагмент сервісу проведення платежу: перед викликом платіжного шлюзу встановлюємо контекст Sentry (tags і extra); при будь-якому exception цей контекст потрапляє в подію. Код строго типізований, без витоку конфіденційних даних у tags (лише ідентифікатори та коди).

**Laravel - сервіс з контекстом Sentry:**

```php
<?php

declare(strict_types=1);

namespace App\Services\Payment;

use App\Exceptions\PaymentGatewayException;
use Illuminate\Support\Facades\Log;
use Throwable;

final class PaymentCaptureService
{
    public function __construct(
        private PaymentGatewayClient $gateway,
        private PaymentRepository $payments,
    ) {}

    /**
     * Спроба провести платіж через шлюз. При неочікуваній помилці - лог + Sentry з контекстом.
     */
    public function capture(int $paymentId, int $userId, string $idempotencyKey): void
    {
        $payment = $this->payments->findOrFail($paymentId);

        $this->pushSentryContext($paymentId, $userId, $payment->account_id, 'capture');

        try {
            $this->gateway->capture($payment->gateway_payment_id, $payment->amount, $idempotencyKey);
            $this->payments->markAsCaptured($paymentId);
        } catch (PaymentGatewayException $e) {
            Log::warning('Payment gateway refused capture', [
                'payment_id' => $paymentId,
                'gateway_code' => $e->getGatewayCode(),
            ]);
            throw $e;
        } catch (Throwable $e) {
            Log::error('Unexpected error during payment capture', [
                'payment_id' => $paymentId,
                'user_id' => $userId,
            ]);
            \Sentry\captureException($e);
            throw $e;
        }
    }

    private function pushSentryContext(
        int $paymentId,
        int $userId,
        int $accountId,
        string $action,
    ): void {
        \Sentry\configureScope(function (\Sentry\State\Scope $scope) use ($paymentId, $userId, $accountId, $action): void {
            $scope->setTag('module', 'payments');
            $scope->setTag('action', $action);
            $scope->setUser(['id' => (string) $userId]);
            $scope->setExtra('payment_id', $paymentId);
            $scope->setExtra('account_id', $accountId);
        });
    }
}
```

**Symfony - аналог (додавання контексту перед викликом шлюзу):**

```php
<?php

declare(strict_types=1);

namespace App\Service\Payment;

use App\Exception\PaymentGatewayException;
use Psr\Log\LoggerInterface;
use Sentry\State\Scope;
use Throwable;

final class PaymentCaptureService
{
    public function __construct(
        private PaymentGatewayClient $gateway,
        private PaymentRepository $payments,
        private LoggerInterface $logger,
    ) {}

    public function capture(int $paymentId, int $userId, string $idempotencyKey): void
    {
        $payment = $this->payments->findOrFail($paymentId);

        $this->pushSentryContext($paymentId, $userId, $payment->getAccountId(), 'capture');

        try {
            $this->gateway->capture($payment->getGatewayPaymentId(), $payment->getAmount(), $idempotencyKey);
            $this->payments->markAsCaptured($paymentId);
        } catch (PaymentGatewayException $e) {
            $this->logger->warning('Payment gateway refused capture', [
                'payment_id' => $paymentId,
                'gateway_code' => $e->getGatewayCode(),
            ]);
            throw $e;
        } catch (Throwable $e) {
            $this->logger->error('Unexpected error during payment capture', [
                'payment_id' => $paymentId,
                'user_id' => $userId,
            ]);
            \Sentry\captureException($e);
            throw $e;
        }
    }

    private function pushSentryContext(int $paymentId, int $userId, int $accountId, string $action): void
    {
        \Sentry\configureScope(function (Scope $scope) use ($paymentId, $userId, $accountId, $action): void {
            $scope->setTag('module', 'payments');
            $scope->setTag('action', $action);
            $scope->setUser(['id' => (string) $userId]);
            $scope->setExtra('payment_id', $paymentId);
            $scope->setExtra('account_id', $accountId);
        });
    }
}
```

У обох прикладах: контрольована відмова шлюзу (`PaymentGatewayException`) лише логується та пробрасывается далі - її можна не відправляти в Sentry, якщо вона очікувана. Неочікувані `Throwable` логуються та явно відправляються в Sentry з уже встановленим контекстом (module, action, user, payment_id, account_id), щоб у інтерфейсі одразу було зрозуміло, який платіж і клієнт постраждали.



## **22. Практична частина уроку**

Усі завдання - у **фінансовому** контексті.

### **Завдання на уроці**

1. **Критичні помилки**  
   Проаналізуйте свій платіжний модуль: які помилки можуть бути критичними (exception при проведенні платежу, при оновленні балансу, при виклику платіжного шлюзу). Визначте, які з них мають потрапляти в Sentry (неочікувані, необроблені або оброблені, але критичні), а які ні (очікувана валідація, контрольована відмова «недостатньо коштів»).

2. **Контекст для помилок**  
   Запропонуйте мінімальний набір контексту для кожної події в Sentry у платіжному потоці: user_id, payment_id, account_id, endpoint, correlation_id, amount (якщо безпечно), gateway_response (код або тип помилки). Опишіть, як би ви додавали цей контекст у коді (middleware для request_id, сервіс платежів перед викликом шлюзу - tags/extra). Код з типізацією: наприклад, метод `reportPaymentFailure(int $paymentId, int $userId, string $gatewayCode): void`, який викликає `Sentry::captureException()` з контекстом.

3. **Інтеграція Sentry (опційно)**  
   У Laravel-проєкті встановіть `sentry/sentry-laravel`, додайте DSN у `.env`, опційно виключіть з відправки `ValidationException` та 404. Переконайтеся, що `APP_ENV` передається в Sentry. Штучно викликайте необроблений exception у тестовому ендпоінті (наприклад, GET /api/v1/test-sentry) і перевірте, що подія з’явилася в Sentry. Додайте контекст (tag `module=payments`, extra `payment_id=1`) перед викиданням - переконайтеся, що він відображається в події.

4. **Сценарій інциденту**  
   Опишіть сценарій: «Після деплою оновлення платіжного модуля користувачі почали повідомляти про помилки при оплаті». Як це виглядає в Sentry (новий issue, збільшення частоти існуючого, прив’язка до release)? Які кроки виконує команда: перегляд issue, контекст (payment_id, gateway), пошук у логах за correlation_id, визначення причини (код, зовнішній API), hotfix або rollback, post-mortem.

5. **Підсумок**  
   Сформулюйте: чим Sentry доповнює звичайні логи; чому контекст (payment_id, user_id) критичний саме для фінансового модуля.
