## БЛОК 2: Навіщо документація і чому OpenAPI

---

### Слайд 4 - API без документації
**На екрані:**
- Заголовок: «API без документації - напівпродукт»
- Проблеми:
  - клієнти вгадують формат по коду чи “усно”;
  - помилки інтеграції через непорозуміння;
  - страх змінювати API, бо ніхто не впевнений, хто чим користується.

**Заметки докладчика:**  
Згадайте реальні історії, коли «бекенд оновили - фронт впав» через відсутність чіткого опису.

---

### Слайд 5 - README vs OpenAPI
**На екрані:**
- Заголовок: «README - це не контракт»
- Погано:
  - текстові описи в README/Notion/чатах;
  - відсутність машинозчитуваного формату.
- Добре:
  - OpenAPI (YAML/JSON) як єдине джерело правди;
  - можливість генерувати UI, клієнтів, тести.

**Заметки докладчика:**  
Поясніть, що хороша документація повинна бути **частиною репозиторію й CI‑процесу**, а не «слайдом у презентації».

---

### Слайд 6 - Що таке OpenAPI/Swagger
**На екрані:**
- Заголовок: «OpenAPI - стандарт, Swagger - інструменти»
- Ідеї:
  - OpenAPI - формат опису REST API (paths, components, schemas...);
  - Swagger UI/Editor - інструменти для перегляду й редагування;
  - описується все: ендпоінти, параметри, тіла, відповіді, помилки, auth.

**Заметки докладчика:**  
Поясніть, що OpenAPI - «JSON/ YAML про ваш JSON/HTTP» - контракт у форматі, який розуміють люди й машини.

## БЛОК 5: HTTP / Feature tests - тести контракту

---

### Слайд 10 - Чому мало лише unit‑тестів
**На екрані:**
- Заголовок: «Unit‑тести ≠ перевірка API»
- Проблема:
  - unit‑тести знають внутрішню реалізацію;
  - вони не ловлять зміни статусів і структури JSON;
  - клієнту байдуже, які сервіси під капотом - його цікавить контракт.

**Заметки докладчика:**  
Поясніть, що API‑тести - це погляд зовні, як у реального клієнта.

---

### Слайд 11 - Що таке Feature / HTTP tests у Laravel
**На екрані:**
- Заголовок: «HTTP‑тести - API з точки зору клієнта»
- Тези:
  - використовують `$this->get()`, `$this->postJson()`, ...;
  - перевіряють статуси, структуру JSON, іноді БД;
  - не знають деталей реалізації.

**Заметки докладчика:**  
Звʼяжіть із «чорною скринькою»: вхід - HTTP‑запит, вихід - HTTP‑відповідь.

---

### Слайд 12 - Що мають перевіряти API‑тести
**На екрані:**
- Заголовок: «Що саме має бути в assert’ах»
- Пункти:
  - статус‑коди (200/201/204/422/404/401/403);
  - структура JSON (`data`, обов’язкові поля, типи);
  - помилки валідації (422 + `errors`);
  - edge‑cases (порожній список, неіснуючий id).

**Заметки докладчика:**  
Поясніть, що перевірка лише `assertStatus(200)` - замало.

---

### Слайд 12a - Практичний приклад: Блок 5 (Feature‑тест успішного створення платежу)
**На екрані:**
- Заголовок: «Практика: тестуємо happy path POST /api/v1/payments»
- Підзаголовок: «201 + структура `data`»
- **→ Демонстрація:** перейти до **lesson_7_3_practice.md**, розділ **«3. Блок 5: Feature test create payment → Слайд 12a»**. Показати:
  - Feature‑тест з `postJson`, підготовкою БД (`RefreshDatabase`);
  - `assertStatus(201)` + `assertJsonStructure`/`assertJsonPath`.

**Заметки докладчика:**  
Поясніть, як цей тест напряму відображає те, що ви записали в OpenAPI.

## БЛОК 6: Тести для помилок (422, 404, 401/403)

---

### Слайд 13 - Тести не лише для «happy path»
**На екрані:**
- Заголовок: «Без тестів на помилки контракт неповний»
- Приклади:
  - невалідні дані (422);
  - неіснуючий ресурс (404);
  - відсутній/невалідний токен (401/403).

**Заметки докладчика:**  
Поясніть, що саме ці ситуації найчастіше “кусали” фронтенд у продакшені.

---

### Слайд 14 - Як тестувати помилки
**На екрані:**
- Заголовок: «Що перевіряти в error‑кейсах»
- Пункти:
  - статус‑код (422/404/401/403);
  - структура `errors` (для 422);
  - поле `message`/`code` для бізнес‑помилок;
  - відсутність HTML‑сторінок і stack trace в JSON.

**Заметки докладчика:**  
Наголосіть, що тести мають гарантувати, що API **ніколи** не віддає HTML/stack trace в продакшені.

---

### Слайд 14a - Практичний приклад: Блок 6 (Feature‑тести 422 і 404)
**На екрані:**
- Заголовок: «Практика: тестуємо 422 і 404 для payments»
- Підзаголовок: «Некоректний POST і GET неіснуючого платежу»
- **→ Демонстрація:** перейти до **lesson_7_3_practice.md**, розділ **«4. Блок 6: Feature tests errors → Слайд 14a»**. Показати:
  - тест для невалідного POST (422 + структура `errors`);
  - тест для `getJson('/api/v1/payments/999999')` (404).

**Заметки докладчика:**  
Поясніть, як такі тести захищають API від «тихих» змін формату помилок.

## БЛОК 7: Документація + тести + CI

---

### Слайд 15 - API‑тести як захист від регресій
**На екрані:**
- Заголовок: «Як тести ловлять зміни контракту»
- Ідеї:
  - змінили статус або назву поля → падає тест;
  - рефакторинг без зміни контракту → тести залишаються зеленими;
  - CI запускає тести на кожен push/PR.

**Заметки докладчика:**  
Поясніть, що це дає команді сміливість змінювати реалізацію без страху поламати клієнтів.

---

### Слайд 16 - Документація + тести = контракт
**На екрані:**
- Заголовок: «Два боки однієї медалі»
- Формула:
  - документація (OpenAPI) - **що має бути**;
  - тести - **що є насправді**;
  - обидва мають оновлюватися при зміні API.

**Заметки докладчика:**  
Попросіть учасників запам’ятати правило: «змінив ендпоінт - обнови OpenAPI і тести».

---

### Слайд 17 - Середовище та запуск Feature‑тестів
**На екрані:**
- Заголовок: «Тести повинні бігати там же, де й застосунок»
- Пункти:
  - запуск у Docker: `docker compose exec app php artisan test --testsuite=Feature`;
  - окрема тестова БД (`.env.testing` / `phpunit.xml`);
  - однакові результати локально й у CI.

**Заметки докладчика:**  
Поясніть, що флейкі‑тести часто пов’язані з тим, що локальне середовище відрізняється від CI.

---
