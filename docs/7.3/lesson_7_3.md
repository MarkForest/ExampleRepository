# **Модуль 7. Проектування та побудова API**

## **Урок 3. Документування API та автоматизоване тестування (HTTP / Feature tests)**



У **Уроці 7.1** ви спроєктували REST-ендпоінти для платежів та рахунків, у **Уроці 7.2** - ввели DTO, FormRequest, Resources та стандартизовані помилки. Контракт API став явним: що приймаємо, що повертаємо, які статуси. Але контракт потрібно **задокументувати** і **перевіряти автоматично**. Інакше клієнти (frontend, мобільний додаток, партнери) не знають точно, як викликати API, а зміни в коді можуть ламати очікування без помітного тесту. У цьому уроці ми розглядаємо **документування API** за стандартом **OpenAPI (Swagger)** та **автоматизоване тестування** API через **HTTP / Feature tests**: тести виконують реальні запити до ендпоінтів і перевіряють статуси та JSON. Усе в контексті **фінансового API** (платежі, рахунки); код тестів - **строго типізований**. Після Модуля 7 ви перейдете до **Модуля 8** (логування та моніторинг): добре задокументоване та покрите тестами API полегшує діагностику проблем у продакшені.



## **1. Мета уроку**

Мета цього уроку - навчити замикати контракт API **документацією** та **автотестами**.

Після уроку ви повинні:

* розуміти, навіщо API-документація критично важлива для інтеграцій та підтримки;
* знати принципи OpenAPI (Swagger): опис ендпоінтів, параметрів, відповідей, помилок;
* вміти описувати ендпоінти та DTO в OpenAPI (вручну або з інструментів);
* писати HTTP / Feature тести для API в Laravel;
* тестувати API як «чорну скриньку» (статус, структура JSON, помилки валідації та 404);
* розуміти, як документація і тести разом захищають контракт і зменшують регресії.



## **2. API без документації - напівпродукт**

Навіть добре реалізоване API марне, якщо ніхто не знає точно, які параметри передавати, який формат відповіді та які помилки можливі. Розробники постійно звертаються до бекенду або вгадують по коду; інтеграції ламаються після змін. Реальність: **API без актуальної документації = API, яким бояться користуватися**. У фінансових інтеграціях (платіжні шлюзи, звіти) помилки через непорозуміння контракту коштують дорого. Документація має бути формалізованою (наприклад, OpenAPI), єдиною та оновлюваною разом із кодом.



## **3. Документація API - це не лише README**

Поганий підхід: текстовий опис у README, «питайте бекендера», приклади в чаті. Така документація швидко застаріває і не дає машинізованої перевірки. Хороший підхід: **формалізований опис** (OpenAPI YAML/JSON), єдиний стандарт, можливість автогенерації клієнтів та UI для перегляду (Swagger UI). Актуальність підтримують за рахунок дисципліни (оновлювати опис при зміні API) або інструментів, що генерують OpenAPI з коду або тестів.



## **4. OpenAPI / Swagger - що це**

**OpenAPI** - це відкритий стандарт опису REST API (формат YAML або JSON). У ньому описують: сервер та базовий шлях, ендпоінти, HTTP-методи, параметри запиту та тіло, відповіді (200, 201, 400, 422 тощо), схеми даних (DTO), авторизацію. **Swagger** - набір інструментів навколо OpenAPI: **Swagger UI** (веб-інтерфейс для перегляду та виконання запитів), **Swagger Editor** (редагування специфікації), генератори клієнтів і серверів. У проєкті зберігають файл `openapi.yaml` або `openapi.json`; його можна віддавати з застосунку або підключати до Swagger UI.



## **5. Навіщо OpenAPI у реальних проєктах**

OpenAPI дозволяє: frontend або партнери працювати з API без постійних уточнень; швидко інтегрувати зовнішні сервіси; автоматично генерувати клієнтський код (TypeScript, PHP тощо); уникати непорозумінь щодо форматів та кодів помилок. OpenAPI - це **договір**: документ описує, як API поводиться. Якщо його підтримувати в актуальному стані, він стає єдиним джерелом правди для розробників і інструментів. Для фінансового API це особливо важливо: партнери та внутрішні клієнти мають покладатися на стабільний опис контракту.



## **6. Що повинна містити хороша API-документація**

Мінімальний набір:

* список ендпоінтів з URL та методами;
* параметри запиту (path, query, body): назви, типи, обов’язковість, приклади;
* приклади запитів (тіло для POST/PATCH);
* відповіді: успіх (200/201) зі схемою даних; помилки (400, 401, 403, 404, 422, 500) з описом та прикладом тіла;
* опис авторизації (Bearer, API key), якщо використовується.

Для кожного ресурсу (наприклад, Payments) описують операції CRUD та відповідні схеми (CreatePaymentRequest, PaymentResponse тощо).



## **7. Документування REST-ресурсів**

Для кожного ресурсу в OpenAPI описують:

* **Операції**: GET /payments (список), GET /payments/{id}, POST /payments, PATCH /payments/{id}, DELETE /payments/{id}.
* **Параметри**: path (id), query (filter, page), body (для POST/PATCH).
* **Відповіді**: 200 зі схемою Payment або списку; 201 зі схемою створеного платежу; 404, 422 зі схемою помилки.

Приклад фрагмента OpenAPI для одного ендпоінта (ідея):

```yaml
paths:
  /api/v1/payments:
    post:
      summary: Create payment
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/CreatePaymentRequest'
      responses:
        '201':
          description: Created
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Payment'
        '422':
          description: Validation error
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/ValidationError'
```

Схеми CreatePaymentRequest, Payment, ValidationError описують у `components/schemas`.



## **8. Документування DTO та помилок**

DTO (вхідні та вихідні) - ключовий елемент документації. У OpenAPI їх описують у `components/schemas`: поля, типи (string, number, integer), обов’язковість, обмеження (min, max, pattern), приклади. Якщо DTO не задокументовані, клієнти не знають точно структуру і API стає нестабільним. Помилки також описують як схеми: загальний вигляд (message, errors або code) для 422, 400, 404 тощо. Це критично для frontend та інтеграцій: вони мають знати, яку структуру парсити при помилці.



## **9. Swagger UI як інструмент комунікації**

Swagger UI показує API в браузері: список операцій, форми для введення параметрів та тіла, кнопку «Execute» та відображення відповіді. Це зменшує кількість питань «як викликати ендпоінт» і прискорює розробку та тестування. Swagger UI не «для галочки» - це робочий інструмент для перевірки контракту та налагодження запитів. У Laravel можна підключити пакет (наприклад, L5-Swagger або аналог) або віддавати статичний Swagger UI з завантаженням вашого `openapi.yaml`.



## **10. Типові помилки в API-документації**

* Документація не відповідає коду - після зміни API опис не оновлено.
* Відсутні приклади тіла запиту та відповіді - складно зрозуміти формат.
* Не описані помилки та їх формат - клієнт не знає, як обробляти 422, 404.
* Різні формати відповідей в описі та в реальності - втрата довіри до документації.
* Документація оновлюється «потім» - вона стає мертвою. Підхід «документація = контракт» вимагає оновлювати її разом із кодом або тестами.



## **11. Автоматизовані тести для API**

Документація описує **як API має працювати**; тести перевіряють, **що воно справді так працює**. Якщо після зміни коду тести проходять - контракт збережено; якщо тест впав - або виправляємо код, або свідомо змінюємо контракт і оновлюємо тест та документацію. Таким чином тести та документація разом захищають API від регресій і розходянь між описом та реалізацією.



## **12. Що таке HTTP / Feature tests**

**Feature tests** (у Laravel - тести, що використовують `$this->get()`, `$this->postJson()` тощо) тестують застосунок як **клієнт**: виконують HTTP-запити до маршрутів і перевіряють статус-код та тіло відповіді (JSON). Вони **не знають** внутрішньої реалізації (які сервіси викликаються); вони перевіряють лише зовнішню поведінку. Це тестування «чорної скриньки»: контракт ззовні. Такі тести ловлять зміни, що ламають клієнтів: зміна назви поля, зміна статус-коду, зміна формату помилки.



## **13. Чим API-тести відрізняються від unit-тестів**

Unit-тести (Модуль 6) тестують окремі класи з моками; вони знають внутрішню логіку. API (Feature) тести не знають реалізацію - лише URL, метод, тіло запиту та очікувану відповідь. Вони максимально близькі до реального клієнта: перевіряють маршрути, middleware, валідацію, формат JSON. У проєкті потрібні обидва: unit - для сервісів і правил, Feature - для контракту API.



## **14. Що повинні перевіряти API-тести**

API-тести перевіряють:

* **Статус-коди**: 200, 201, 204 для успіху; 422 для валідації; 404 для відсутнього ресурсу; 401/403 для доступу.
* **Структуру JSON**: наявність очікуваних полів, тип значень (наприклад, `data.id` - integer).
* **Помилки валідації**: при невалідних даних - 422 та структура `errors` по полях.
* **Edge cases**: порожній список, неіснуючий id, заборонений доступ.

Не перевіряють внутрішню логіку сервісів - лише **поведінку API** згідно з контрактом.



## **15. Структура хорошого API-тесту**

Кожен тест зазвичай має:

1. **Підготовку** - створення тестових даних (користувач, рахунок, токен при потребі); використання `RefreshDatabase`, щоб не чіпати dev-базу.
2. **Виконання запиту** - `$response = $this->postJson('/api/v1/payments', $payload);`.
3. **Перевірку статусу** - `$response->assertStatus(201);`.
4. **Перевірку JSON** - `$response->assertJsonStructure(['data' => ['id', 'amount', 'currency']]);` або `assertJsonPath('data.amount', '100.00');`.
5. За потреби - перевірку стану БД (наприклад, що платіж з’явився в таблиці з очікуваними полями).

У Laravel використовують `$this->get()`, `$this->postJson()`, `$this->patchJson()`, `$this->delete()` тощо; для API зазвичай `*Json()` та `assertJson*`.



## **16. Тестування помилок**

Обов’язково тестувати не лише «щасливий шлях», а й помилки:

* **Некоректні дані** - відсутнє обов’язкове поле, невірний тип - очікуємо 422 та структуру помилок.
* **Неіснуючий ресурс** - GET /payments/99999 - очікуємо 404.
* **Заборонений доступ** - запит без токена або з невалідним - 401 або 403.

API без тестів на помилки небезпечне: клієнти можуть отримувати неочікувані статуси або формати. У фінансовому API важливо перевіряти, що при «недостатньо коштів» повертається передбачуваний код та повідомлення.



## **17. API-тести як захист від регресій**

Після додавання нової фічі або рефакторингу запуск Feature-тестів показує, чи не зламався контракт: чи залишилися ті самі статуси та структура JSON. Якщо хтось випадково змінить назву поля або формат помилки - тест впаде. Це дозволяє рефакторити без страху і захищає frontend та інтеграції від неочікуваних змін. Тести можна запускати в CI та перед мерджем; у Docker - `docker compose exec app php artisan test --testsuite=Feature`.



## **18. Документація + тести = контракт**

Разом документація (OpenAPI) і тести забезпечують контракт: документація каже **що має бути**, тести перевіряють **що є**. Якщо тести проходять - реалізація відповідає очікуваній поведінці; якщо оновлюємо контракт - оновлюємо і опис, і тести. Це зменшує кількість багів інтеграції та питань до бекенду і підвищує довіру до API. У фінансових проєктах такий підхід особливо цінний: контракт явний, зміни контрольовані.



## **19. Типові помилки при тестуванні API**

* Тестування лише «happy path» - помилки залишаються непокритими.
* Відсутність перевірок помилок (422, 404) - клієнт може отримати неочікувану відповідь.
* Перевірка лише статусу без JSON - структура може змінитися, тест не вловить.
* Тести, залежні від порядку виконання або глобального стану - потрібна ізоляція (RefreshDatabase, окремі дані в кожному тесті).
* Відсутність підготовки тестових даних - нестабільні тести при зміні БД.



## **20. Середовище та запуск тестів**

Проєкт, як і в попередніх модулях, розгортається через **docker-compose**. Feature-тести запускають у тому ж середовищі: `docker compose exec app php artisan test` або `php artisan test --testsuite=Feature`. База для тестів налаштовується через `.env.testing` або `phpunit.xml` (наприклад, SQLite в пам’яті або окрема тестова MySQL). Єдине середовище забезпечує однакову поведінку тестів у всіх розробників та в CI.



## **21. Практична частина уроку**

Усі завдання - у **фінансовому** контексті, зі **строгою типізацією** у коді тестів.

### **Завдання на уроці**

1. **Опис API в OpenAPI**  
   Візьміть ресурс Payments (або Accounts) з вашого API. Опишіть у OpenAPI (YAML або JSON): ендпоінти (GET list, GET one, POST, PATCH, DELETE), схеми CreatePaymentRequest та Payment (відповідь), відповіді 201, 422, 404. Додайте приклад тіла запиту для POST та приклад відповіді 422.

2. **Swagger UI (опційно)**  
   Підключіть Swagger UI до проєкту (пакет або статичний build) і завантажте ваш OpenAPI-файл. Перевірте один ендпоінт через UI (наприклад, GET /api/v1/payments) і переконайтеся, що відповідь відповідає опису.

3. **Feature-тест: успішне створення платежу**  
   Напишіть тест: підготовка даних (користувач, рахунок; при потребі авторизація), `postJson('/api/v1/payments', $payload)` з валідними даними. Перевірте статус 201, наявність у відповіді полів (наприклад, `data.id`, `data.amount`, `data.currency`). Використовуйте `RefreshDatabase` та типізацію.

4. **Feature-тест: помилка валідації та 404**  
   Тест валідації: відправити POST з невалідними даними (наприклад, від’ємна сума або відсутнє поле). Перевірити статус 422 та структуру `errors`. Тест 404: `getJson('/api/v1/payments/99999')` - очікувати 404.

5. **Відповідність документації та тестів**  
   Переконайтеся, що описані в OpenAPI статуси та схеми відповіді збігаються з тим, що перевіряють тести. Коротко опишіть: як ви будете підтримувати документацію в актуальному стані при зміні API (наприклад, правило «змінив ендпоінт - оновив OpenAPI і тести»).
